# Артефакт — музей размером с город

Сайт-проект: городской музей, где экспонаты расположены по всему Томску (и другим городам).

## Файлы

- `index.html` — публичный сайт. Весь HTML, CSS и JS в одном файле.
- `admin.html` — панель управления контентом. Тоже self-contained.
- `Images/` — фоновые изображения для hero-секции (Sky.png, City.png).
- `photos/` — реальные фото экспонатов, организованы по папкам (`photos/название-экспоната/`).

## Архитектура данных

Все данные сайта хранятся прямо внутри `index.html` между маркерами:
```
/* __SITE_DATA_START__ */
const SITE_DATA = { ... };
/* __SITE_DATA_END__ */
```

`SITE_DATA` содержит: `artifacts[]`, `about`, `manifest`, `contacts`, `authors[]`, `settings`.

**Синк index.html ↔ admin.html:**
- `index.html` при каждой загрузке пишет `SITE_DATA` в `localStorage('artefact-site')`
- `admin.html` читает из `localStorage` и редактирует там
- При публикации admin берёт локальный `index.html` как шаблон, вставляет новый `SITE_DATA`, пушит на GitHub
- Итог: открыл `index.html` → обновился localStorage → admin видит актуальные данные автоматически

## Структура index.html

Навигация между разделами через функцию `navigate(view, id)`:
- `view-main` — галерея + hero-блок
- `view-about` — о музее (факты + манифест)
- `view-artifact` — карточка экспоната (фотослайдер + мини-карта)
- `view-authors` — список авторов (view-contacts удалён из навигации)
- `view-author` — страница автора
- `view-map` — общая карта всех экспонатов

`navigateToGallery()` — отдельная функция, якорный скролл к gallery-section минуя hero.

## Стек

- Чистый HTML/CSS/JS, никаких фреймворков и бандлеров
- **MapLibre GL JS 4** для карт (OpenFreeMap positron, без API-токена)
- Шрифт Inter (Google Fonts)
- Всё в одном файле — легко деплоить как статику

## Стиль кода

- CSS-переменные в `:root` (`--black`, `--white`, `--border`, `--edge`, `--nav-h`)
- Функции в глобальном скоупе
- Комментарии-разделители вида `/* ── SECTION ──────── */`

## Экономия контекста — ОБЯЗАТЕЛЬНО

`index.html` — монолит ~2200 строк (HTML+CSS+JS+инлайн-SVG). Чтение целиком раздувает контекст и приводит к ошибке `1M context`.

- **Никогда не читать `index.html` целиком.** Сначала `grep -n` по якорю, потом `Read` с `offset`/`limit` на нужный фрагмент.
- **Не читать фото/бинарники** (`photos/`, `Images/`) — они в `.claudeignore` и заблокированы в `.claude/settings.json`. Пользователь сам присылает скриншоты.
- **Длинная сессия — главный источник раздувания.** После крупного блока работы предлагать пользователю `/compact` (сжать историю) или `/clear` + новая сессия под новую задачу.
- При правках через `Edit` не перечитывать файл для проверки — Edit сам падает если `old_string` не найден.

## Важные технические детали

- **MapLibre координаты**: порядок `[lng, lat]` — не путать с Leaflet (`[lat, lng]`)
- **Hero-анимация**: JS-параллакс владеет `style.transform` на `.hero-layer`. CSS-анимации применяются к inner-SVG элементам (`.hero-artefact-svg` и т.д.), иначе конфликт
- **Логотип**: SVG встроен инлайн в `.nav-logo-main` с `fill="currentColor"`. `applySettings()` не должен перезаписывать его текстом — только если загружена кастомная картинка
- **logoSub**: хранится с `\n`, рендерится через `white-space: pre-line` + `ls.textContent = s.logoSub`
- **Автор неизвестен**: не добавляется в массив `authors[]` — только опция в select админки. На странице экспоната отображается как plain text без ссылки (стандартное поведение когда `authorObj` не найден)
- **Фото**: PNG/HEIC конвертируется через `sips`, качество 75, максимум 1600px. Экспонаты: `photos/название-папки/1-cover.jpg`, `2.jpg`… Авторы: `photos/hudozhniki/имя-фамилия.jpg`. Папки: латинский транслит, строчные, дефис.
- **Bio автора**: хранится с `\n\n`, рендерится через `white-space: pre-line` + `el.textContent = s.bio` (как logoSub)

## Само-аудит — постоянный процесс, БЕЗ напоминания

Это не опция и не реакция на просьбу пользователя. Это часть каждой задачи.

### После каждого блока работы
1. Нетривиальная ошибка или неожиданная находка? → сразу в `.claude/rules/learned.md`
2. Изменилась архитектура/CSS-паттерны/структура страниц? → обновить `.claude/commands/site.md`
3. Затронул элемент с окружением (padding, display, z-index)? → проверить соседей визуально

### Периодически (раз в ~5 сессий или по запросу)
- Удалить мусорные файлы (`.DS_Store`, test-*, неиспользуемые скрипты)
- Проверить `settings.local.json`: только паттерны, никаких одноразовых команд с путями
- Сверить скилл-файлы с реальным состоянием кода

**Записывать в Learned:** баги от неверных предположений о браузере/CSS/JS, конфликты механизмов, паттерны экономящие время
**Не записывать:** опечатки, добавление контента, рутинные правки

**`settings.local.json`:** только паттерны (`Bash(sips *)`, `Bash(grep *)`). Одноразовое → одобрить интерактивно, не сохранять. Список > 25 строк → почистить.

## Доступность и юзабилити

### WCAG 2.1 AA — обязательные правила
- Обычный текст: контраст ≥ **4.5:1** (opacity на чёрном по белому ≥ **0.55**)
- Крупный текст (≥ 18pt): контраст ≥ **3:1**
- Цвет не единственный способ передачи информации
- Все интерактивные элементы доступны с клавиатуры + видимый focus indicator
- Текст масштабируется до 200% без потери функциональности
- Контент переотображается при 320px без горизонтального скролла

### Эвристики Нильсена
1. **Статус системы** — пользователь всегда знает что происходит
2. **Язык пользователя** — без жаргона, понятные слова
3. **Свобода действий** — явная кнопка «назад» / отмена
4. **Согласованность** — одни и те же слова означают одно и то же везде
5. **Предотвращение ошибок** — лучше не дать совершить ошибку
6. **Узнавание > воспоминание** — показывай варианты, не заставляй помнить
7. **Гибкость** — ускорители для опытных, не мешающие новичкам
8. **Минимализм** — каждый лишний элемент конкурирует с нужным
9. **Помощь при ошибках** — понятный язык + конкретное решение
10. **Документация** — если нужна, легко находится и ориентирована на задачу

### Протокол критических изменений
Перед изменениями затрагивающими архитектуру, навигацию, несколько разделов сразу или необратимыми — **описать план и спросить подтверждение**. Простые правки (текст, цвет, новый экспонат) — делать сразу.

## Стратегия сабагентов

Использовать `Agent` когда задача требует широкого чтения или изолированного контекста:

| Когда | Тип агента |
|-------|-----------|
| Аудит синка admin↔index (читать 2 больших файла) | `general-purpose` |
| Конвертация/оптимизация пачки фото | `general-purpose` |
| Проверка безопасности всего JS | `general-purpose` |
| Поиск символа или паттерна в коде | `Explore` |

Простые правки CSS/текста/одного блока — делать напрямую, без сабагентов.
